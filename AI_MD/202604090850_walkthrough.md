# WebSocket 블록 이벤트 순차 처리(Queue) 도입 결과

## 변경 개요 
기존에는 수신된 `block` 이벤트를 즉각적으로 `await handleNewBlock()`으로 호출하여 잦은 빈도의 이벤트 유입 시 동시성 충돌이나 블록 꼬임 현상이 발생할 가능성이 있었습니다. 
이번 변경을 통해 NodeJS의 비동기 단일 스레드 구조를 활용한 메모리 큐(Queue) 패턴을 도입해 병목 없는 안정적인 실시간 적재가 이뤄지게 하였습니다.

## 작업 내역 및 변경 사항
1. **메모리 큐 상태 변수 추가** (`src/block/service/block.service.ts`)
   - `blockQueue: number[] = []` : 들어온 이벤트를 차곡차곡 쌓아놓는 인메모리 배열
   - `isProcessingQueue = false` : 워커 작동 중복を防지하는 세마포어(상태 플래그)

2. **WebSocket 구독 리스너 변경**
   ```typescript
   this.wsService.onEvent('block', async (...args: any[]) => {
     const blockNumber = args[0] as number;
     this.logger.debug(`[WS Event] 블록 ${blockNumber} 큐에 적재`);
     
     // 이벤트를 큐에만 추가하고 비동기 워커를 실행요청 (블로킹하지 않음)
     this.blockQueue.push(blockNumber);
     this.processBlockQueue();
   });
   ```

3. **비동기 큐 프로세싱 메서드 신규 추가**
   ```typescript
   private async processBlockQueue() {
     // 현재 이미 큐를 소화 중이라면 중복 실행 무시
     if (this.isProcessingQueue) return;
     this.isProcessingQueue = true;

     try {
       // 처리할 큐가 남아있을 때까지 하나씩 꺼내고 기다리며 순차 처리
       while (this.blockQueue.length > 0) {
         const blockNumber = this.blockQueue.shift();
         if (blockNumber) {
           await this.handleNewBlock(blockNumber);
         }
       }
     } catch (error: any) {
       this.logger.error(`블록 큐 처리 중 오류 발생: ${error.message}`);
     } finally {
       // 모든 큐를 처리하면 워커 상태 플래그 초기화
       this.isProcessingQueue = false;
     }
   }
   ```

### 해결된 점
이로 인해 WebSocket 이벤트가 아무리 한꺼번에 도착하더라도, DB I/O를 발생시키는 `handleNewBlock` 메서드는 기존 작업이 완전히 종료된 이 후에만 다음 이벤트를 처리하게 되어 트랜잭션 오류 및 데이터 동시성 꼬임 문제를 완벽하게 예방하게 되었습니다.

## 테스트 방법
현재 설정된 구조에서 PM2 또는 NPM 데브 서버를 실행(`npm run start:dev`)하신 뒤, 블록체인에서 신규 블록이 생성되어 이벤트 로그에 `[WS Event] 블록 XXXX 큐에 적재` 텍스트가 순차적으로 콘솔에 출력됨을 확인할 수 있습니다.
