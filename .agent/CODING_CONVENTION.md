# 코딩 컨벤션 및 스타일 가이드 (Coding Conventions)

본 문서는 프로젝트의 일관된 코드 품질을 유지하고, 효율적인 블록체인 데이터 처리를 위한 필수 가이드라인입니다.

---

## 1. 코드 스타일 및 명명 규칙
- **포맷팅**: 스페이스 2칸, 단일 따옴표, 후행 쉼표 사용 (`.prettierrc`).
- **명명 규칙**: 클래스/인터페이스(`PascalCase`), 변수/메서드(`camelCase`), 파일/폴더(`kebab-case`).
- **엄격한 타입 제약**:
  - `any` 타입 사용을 원칙적으로 금지합니다.
  - 블록체인 원시 데이터(Raw Header, Log 등)는 반드시 전용 인터페이스(예: **`EthBlockHeader`**)를 정의하여 사용해야 합니다.

## 2. 아키텍처 원칙
- **계층 분리**: 비즈니스 로직은 철저히 `Service` 계층에 격리합니다.
- **로깅**: 전역 `WinstonLogger` 만을 주입받아 사용하며, `console.log` 사용을 금지합니다.
- **DTO 유효성 검사**: API 진입 시 `class-validator`를 통한 강제 필터링을 수행합니다.

## 3. 블록체인 연동 특화 정책
- **데이터 정규화**: 해시 및 주소값은 DB 저장 전 필히 `.toLowerCase()`로 정규화합니다.
- **BigInt 처리**: JSON 직렬화 및 DB 저장을 위해 `BigInt` 값은 항상 `.toString()`으로 변환합니다.
- **RPC 에러 대응**: 노드 오류(429 등) 발생 시 지수승 기반의 **Retry(재시도)** 매커니즘을 필수로 적용합니다.
- **네트워크 효율**: 대량 데이터 조회 시 `batchMaxCount`를 고려한 배치 처리를 기본으로 합니다.

## 4. 유닛 테스트 규격
- **테스트 대상**: 비즈니스 로직이 집중된 `Service` 계층의 커버리지를 80% 이상으로 유지합니다.
- **Mocking**: 실제 DB 커넥션을 배제하고 Repository 및 외부 서비스를 철저히 Mocking하여 테스트 속도를 확보합니다.
