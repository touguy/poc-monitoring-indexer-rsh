# Ponder 심화 최적화 기능 (Phase 2) 반영 완료

사용자님의 승인에 따라 Ponder 인덱서의 핵심 기술인 **과거 데이터 대량 동기화(Bulk Chunking)** 및 **로컬 RPC 캐싱 계층(RPC Cache Layer)**을 `poc-monitoring-indexer-rsh`에 성공적으로 도입했습니다.

## 📝 문서(Docs) 업데이트 내역
1. **`PRD.md`**: 옵션 5(Bulk Chunking) 및 옵션 6(RPC Caching) 항목을 Ponder 확장 아키텍처 리스트에 추가 완료.
2. **`.agent/BUSINESS_LOGIC.md`**: 초기 동기화 워커 부분에 `Promise.all` 및 Dynamic Chunking 기믹을 명세하고, 주기적 검증(Polling) 파트에 `LRU` 기반의 캐시 정책을 서술 완료.

## 🛠️ 코드(Code) 구현 내역

### 1. (옵션 6) RPC Cache Layer 구현
- **위치**: `src/blockchain/service/blockchain-rpc.service.ts`
- **구현**: 노드 반복 호출 방지를 위해 `blockCache` 인메모리 Map을 도입했습니다. 메모리 부족(OOM)을 방지하기 위해 사이즈가 `1000`개를 초과하면 들어온 순서대로 삭제(LRU 동작 모방)하도록 간이 로직을 추가했습니다. `safe`나 `latest`와 같은 동적 태그는 매번 값이 달라지므로 캐싱에서 제외했습니다.

### 2. (옵션 5) Bulk Chunking 기반 대량 동기화
- **위치**: `src/block/service/block.service.ts` -> `saveBlocksInRange`
- **구현**: 1000개의 갭이 있다면 예전처럼 1개씩 1000초가 걸리게 동작하지 않습니다. 50개(ChunkSize)의 블록을 `Promise.all`을 이용해 병렬로 긁어옵니다. 노드의 Rate Limit을 방지하기 위해 각 청크 단위로 200ms의 짧은 지연만 줍니다. 수집 속도가 비약적으로 향상되었습니다.

### 3. (옵션 5) Dynamic Chunking 기반 이벤트 재시도
- **위치**: `src/contract/service/contract-event.service.ts` -> `fetchAndSaveEventsForBlock`
- **구현**: 거대한 범위의 이벤트를 수집하다가 노드에서 `Exceeded limit (10000 logs)` 류의 에러가 발생할 경우를 대비하여 `catch` 블록 내에 Dynamic Chunking을 구현했습니다. 범위가 너무 커 에러가 났다면 즉시 범위를 반(mid)으로 쪼개서(`fromBlock`~`mid`, `mid+1`~`toBlock`) 자신을 재귀 호출하며 무결성을 보장합니다.
