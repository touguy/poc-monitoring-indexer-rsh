# Ponder 심화 최적화 기능 (Phase 4) 반영 완료

사용자님의 지시에 따라 WebSocket `newHeads` 구독 시 전달되는 상세 헤더 정보를 활용하여 실시간 인덱싱 성능을 최적화하는 **[옵션 9] WS Block Header 활용 최적화** 기능을 성공적으로 도입했습니다.

## 📝 문서(Docs) 업데이트 내역
- **`PRD.md`**: 옵션 9(WS Block Header 활용 최적화) 항목을 Ponder 확장 기능 리스트에 추가했습니다.
- **`.agent/BUSINESS_LOGIC.md`**: WebSocket 원문 페이로드 파싱을 통해 `eth_getBlockByNumber` 호출을 생략하는 최적화 매커니즘을 명세했습니다.

## 🛠️ 코드(Code) 구현 내역

### 1. `BlockchainWsService` 개선
- **Raw Message Interceptor**: `ethers.WebSocketProvider`의 내부 소켓에 직접 접근하여 JSON-RPC `eth_subscription` 메시지를 가로챕니다.
- **Header Extraction**: `newHeads` 이벤트에서 블록 번호뿐만 아니라 `hash`, `parentHash`, `timestamp`, `logsBloom` 등 핵심 데이터를 추출하여 전달하는 `onBlockHeader()` 메서드를 구현했습니다.

### 2. `BlockService` 최적화
- **Queue Refactoring**: `blockQueue`에 블록 번호와 함께 WS에서 받은 헤더 정보를 함께 담도록 구조를 변경했습니다.
- **RPC Skip Logic**: `handleNewBlock`에서 WS 헤더 정보가 존재할 경우, 별도의 RPC 호출 없이 즉시 데이터를 가공하여 DB 저장 및 Reorg 검증 단계로 진입합니다. 헤더가 누락된 경우에만 기존의 RPC 폴백 로직이 동작합니다.
