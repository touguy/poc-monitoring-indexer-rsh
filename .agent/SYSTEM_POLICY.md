# 시스템 공통 정책 및 프로세스 (SYSTEM_POLICY.md)

애플리케이션 전반에서 사용되는 공통 컴포넌트 정책 및 런타임 구동 시 기반이 되는 환경 변수 세팅 정책입니다. 이 문서의 원칙에 입각해야 전역 예외 처리 및 로깅 패턴의 일관성을 잃지 않을 수 있습니다.

---

## 1. 공통 컴포넌트 정책

- **주석 및 설명 의무화 정책 (AI 통합 코딩 룰)**: 
  - 코드가 신규 추가되거나 수정될 경우, **반드시 모든 함수(Function)와 중요 로직 상단에 한글로 동작 과정과 목적을 설명하는 주석(JSDoc 및 인라인 주석)**을 달아야 하며, 상세한 한글 주석 스타일(예: 핵심 비즈니스 로직, 처리 순서 1~5 명시 등)을 작성해야 합니다.
  
- **의존성 주입 (Dependency Injection)**: 
  - 모든 클래스는 `@Injectable()` 되며 생성자 주입을 필수로 합니다. 소스상에서 직접 `new 클래스()`하여 인스턴스를 무위로 생성하는 것을 엄격히 금지합니다.

- **전역 로깅 (Logging Policy)**: 
  - NestJS 내장 로거 대신 커스텀 `nest-winston` 모듈을 전역 애플리케이션 레벨에 적용합니다. (`main.ts` 에서 `bufferLogs: true` 지시자 사용)
  - 일반 `console.log()`는 개발/디버깅을 포함해 어떤 이유에서건 사용해서는 안 됩니다.
  - 전역 스코프에 환경변수 `APP_NAME`을 태깅하여 로그 포맷팅 시 구분이 쉽도록 시각화합니다.

- **전역 예외 및 응답 처리 전략 (Exception Mapping)**: 
  - 자체 정의된 커스텀 에러 인터페이스인 `BusinessException(ErrorCode)`을 통해 비즈니스 오류를 던지도록 시스템화합니다. 
  - 전역에 마운트된 `GlobalExceptionsFilter`가 예기치 않은 및 의도된 예외를 모두 낚아채 애플리케이션의 통일된 API 응답 규격인 **`ResultResDto.error()`** 형태의 JSON으로 직렬화해 자동 응답 처리합니다. 이로써 외부로 Raw 에러 스택 구조가 노출되지 않도록 철저히 차단합니다.

## 2. 애플리케이션 환경 변수 스펙 (`.env`)

시스템은 부트스트랩 시 및 동작 중에 `@nestjs/config` 모듈의 `ConfigService`를 기반으로 다음 환경 변수를 동적으로 읽어야 합니다.

| 환경 변수명 | 기본값 | 목적 및 설명 |
|---|---|---|
| `APP_NAME` |  | 앱 로깅 네임스페이스 명칭 설정 (예: `monitoring-indexer`) |
| `PORT` | `3000` | 서비스가 동작하고 바인딩할 포트 번호 |
| `DB_HOST` | | PostgreSQL 원격 접속 주소 호스트 |
| `DB_PORT` | | PostgreSQL 원격 접속 포트 (보통 5432) |
| `DB_USER` | | PostgreSQL 접속 계정명 |
| `DB_PASSWORD` | | PostgreSQL 접속 비밀번호 |
| `DB_NAME` | | 프로젝트에서 사용할 타겟 데이터베이스 명 |
| `RPC_URL` | | 블록체인 노드를 순회 조회할 HTTP 기반 풀노드 주소 |
| `WSS_URL` | | 블록 실시간 스트림 이벤트를 감청할 웹소켓 노드 라우터 주소 |
| `TARGET_CONTRACT_ADDRESS` | | 실시간으로 이벤트를 모니터링하고 추적할 대상 스마트 컨트랙트 주소 (예: `0x45...`) |
| `INITIAL_SYNC_BLOCK_COUNT` | `100` | 백엔드 기동 직후 부족한 과거 블록 격차(Gap)를 보완하기 위한 조회 상한 허용치 |
| `POLLING_CRON_TIME` | `*/60 * * * * *` | 주기적 Polling(자동화 검증)을 위한 스케줄러 간격을 정하는 Cron 수식 |
